# AI Bridge v0.4.3 Fresh Thread Plugin Discovery Playbook

## Scope

This playbook separates three independent questions:

1. Can the local AI Bridge MCP server initialize and list its tools?
2. Is the local plugin layout internally consistent?
3. Does a fresh ChatGPT/Codex thread actually expose `ai_bridge_*` tools?

v0.4.3 adds bounded local fact collection and a repeatable manual experiment record. It does not continue real Claude dogfood, create an AI Bridge run, publish a package, create a tag or release, or change MCP/worker runtime behavior.

## Local Diagnostics

Run from the repository root:

```powershell
npm run smoke:mcp-tools
npm run diagnose:plugin
npm run diagnose:codex-discovery
npm run diagnose:codex-discovery -- --json
```

`diagnose:codex-discovery` records versions, MCP entry-point facts, the isolated smoke result, plugin-layout status, explicit environment paths, a fixed list of local directory hints, and path-shape warnings. It performs existence checks only. It does not recursively scan HOME, enumerate the environment, read tokens, or inspect broad logs.

## Fresh Thread Manual Checks

1. Close and reopen the intended ChatGPT/Codex client.
2. Open a genuinely new thread rather than reopening an old thread.
3. Record whether AI Bridge appears as an installable or active plugin.
4. Record whether any `ai_bridge_*` tools are visible to the thread.
5. If tools are visible, call `ai_bridge_discover_workspace_runs` before creating a run.
6. If tools remain absent, retain the local diagnostic outputs and classify the blocker outside AI Bridge MCP runtime.

Do not bypass a missing MCP exposure layer with direct JSON-RPC or shell calls for dogfood.

## Installation / Discovery Hypotheses

These are investigation categories, not asserted Codex rules:

- the repository is not in a client-supported plugin source or installation location;
- the plugin was installed but the client index/cache did not refresh;
- the client supports a different plugin discovery surface than the one used;
- the plugin manifest or `.mcp.json` is not visible to the active client process;
- the current client build does not expose local plugin candidates in the expected UI;
- an existing thread retained an older tool registry after restart.

Directory existence alone does not prove installation, indexing, or active client support.

## Known Limitations

- No local script can prove ChatGPT/Codex UI or thread tool exposure.
- Candidate directory checks are filesystem observations, not documented discovery rules.
- The diagnostic does not read client indexes, caches, tokens, configuration contents, or logs.
- The diagnostic does not continue real Claude dogfood or invoke AI Bridge tools.
- Client restart experiments require manual observation outside the repository.

## Follow-up Decision

Keep real Claude dogfood paused until a fresh Codex thread directly exposes `ai_bridge_*` tools. If all three local diagnostics pass and tools remain absent, retain the JSON report and escalate the remaining issue as Codex/plugin installation, indexing, discovery, client-support, or UI exposure outside AI Bridge runtime.
