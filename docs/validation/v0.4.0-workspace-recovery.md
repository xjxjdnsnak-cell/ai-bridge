# AI Bridge v0.4.0 Workspace Recovery Validation

Date: 2026-06-27

## Scope

v0.4.0 adds workspace-level discovery, attach, and polling on top of the v0.3.5 durable worker. The authoritative state remains each run's `run.json`; the workspace index is an accelerator.

## Requirement Matrix

| Requirement | Implementation | Evidence |
| --- | --- | --- |
| Discover without runId/taskId | `discoverWorkspaceRuns()` | workspace discovery tests |
| Attach running or completed task | `attachWorkspaceRun()` | cross-process running/completed recovery test |
| Poll by workspace | `pollWorkspaceRun()` | workspace poll tests |
| Preserve Claude context | existing run `claudeSessionId` and resume mode | needs_fix iteration 2 test |

| Legacy v0.3.5 compatibility | legacy path matching and fenced lazy backfill | legacy run test |
| Prevent silent duplicate active runs | preflight warning/reuse/explicit concurrency | preflight duplicate test |
| Moved workspace safety | git remote fingerprint and explicit confirmation | moved workspace test |

## Test Matrix

`tests/workspace-recovery.test.mjs` covers:

- equivalent path normalization and Windows case/separator behavior;
- symlink or Windows junction realpath normalization;
- basic discover and attach without creating a run;
- corrupt workspace index scanning and rebuild;
- moved workspace candidate classification and explicit confirmation;
- legacy v0.3.5 discovery and lazy backfill;
- deterministic multiple-run ranking and ambiguity;
- preflight warning, reuse, and explicit concurrent run creation;
- workspace polling without taskId;
- independent-process durable running task recovery and disconnected transcript output;
- completed task recovery with `awaiting_review`, complete finalization, transcript, and final log;
- `needs_fix` continuation with the original Claude session and resume mode.

## CI

The final source SHA and its GitHub Actions run are recorded after push. Both `ubuntu-latest` and `windows-latest` must run `npm run check`, `npm test`, and `npm run test:integration`.

Initial implementation run `28279137039` passed Ubuntu but failed Windows test `new process discovers, attaches, and polls the original durable running task`: the task had reached `completed` while attach still returned a previously read run status of `running`. This was a v0.4.0 workspace recovery race, not the historical terminal test. The fix makes terminal attach call idempotent finalization and reread authoritative run state. The focused failing scenario then passed three consecutive runs and the full suite passed 77/77.

## Known Limitations

- v0.4.0 validation uses fake Claude fixtures and persisted-state process tests. It does not prove real Claude API behavior.
- v0.4.0 provides workspace-level discovery and attach after reopening Codex. It does not guarantee automatic MCP client reconnect or live push replay.
- Moved workspace matching is best-effort and requires explicit confirmation.
- The workspace index is not authoritative; corruption can add diagnostics and scan cost but cannot replace run state.
- Historical Windows terminal-finalization timing remains a watch item. It must not be hidden with retries if it reproduces.
