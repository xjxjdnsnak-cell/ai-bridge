# AI Bridge v0.4.6 In-flight Disconnect Retry

## Scope

This document records a real AI Bridge transport interruption while a local
Claude Code task was still running. The retry validates persisted recovery
through a fresh Codex thread without creating a replacement run, task,
iteration, or Claude session.

This is a validation gate for the existing v0.4.3 implementation. No product
code, tests, scripts, versions, CI configuration, plugin manifest, skills, or
README were changed.

## Why v0.4.5 Failed

v0.4.5 relied on a Claude Bash delay that required interactive approval and
never executed. Claude completed at `2026-06-28T05:44:59.645Z`; the MCP server
was stopped later at `2026-06-28T05:45:19.3846086Z`. The last poll had shown a
stale `running` state, but the authoritative timestamps proved the closure was
post-completion.

v0.4.6 therefore used no artificial delay. It started an information-dense
document task and scheduled a system-side MCP server stop immediately after
the start response.

## Fresh Thread Tool Exposure

workspace discovery first, then attached the original run.

This is direct evidence for these local threads only. It does not prove global
or permanent plugin discovery behavior.

## Non-interactive Hold Strategy

No Bash delay, `node -e`, `sleep`, timeout command, or PowerShell delay was used
to keep Claude active. The intended hold was the requested long-form writing
work itself. An external monitor watched for the new durable worker and, after
detecting it, waited 6 seconds before stopping only MCP server processes.

Claude did violate the pure-text constraint by requesting a Bash directory
check (`ls docs/validation/...`) at transcript event 41. That command was not
the hold mechanism and did not create the in-flight window. The deviation is a
review finding; Claude still modified only the authorized document.

## Disconnect Method

The system-side monitor stopped only AI Bridge `mcp/server.mjs --stdio`
processes:

- MCP server PID `51784`
- MCP server PID `62936`

It did not terminate durable worker PID `37276` or Claude PID `60396`, delete
AI Bridge state, or edit run/task JSON. Immediately after the stop, both the
worker and Claude processes were still running.

## Verdict

**weak pass**

`disconnect completedAt < Claude completedAt`, so the transport closure
occurred in flight. By the time the workspace was reattached, Claude had
completed. This validates persistence across in-flight disconnect and
completed-task recovery after reconnect, but not live running-state polling
after reconnect.

## What Failed Or Was Not Verified

- Recovery did not observe the original task still running, so this is not a
  strong pass.
- Claude requested a Bash directory check despite the pure-text constraint.
- The original client transport did not demonstrate automatic reconnection.
- Live push replay and retroactive output replay were not validated.
- Multiple consecutive real Claude iterations were not tested.
- General plugin discovery reliability was not proven.

## Limitations

- One Windows workspace, one documentation task, and one transport-failure
  method were tested.
- The external monitor created the timing window; other failure modes and task
  durations may behave differently.
- Weak pass proves in-flight persistence followed by completed-task recovery,
  not running-state polling after reconnect.
- No claim is made about automatic reconnect guarantees, live replay
  guarantees, all real Claude API behavior, public distribution, or global
  plugin discovery.

## Follow-up Decision

Accept v0.4.6 as a weak pass. No release-blocking AI Bridge defect was exposed,
so no code or version change is warranted. A future strong-pass retry should
open the recovery thread sooner or use a longer pure text task, while retaining
the same authoritative timestamp comparison and avoiding Claude shell tools.
