## Limitations

- This is one local Windows run and a short documentation task.
- The result proves post-completion transport closure and completed-task
  recovery in a fresh thread.
- It does not prove persistence across an in-flight transport disconnect.
- Claude exited successfully without implementing the requested document, so
  exit code 0 alone was not treated as evidence of task completion.
- No claim is made about all real Claude API behavior, public distribution,
  automatic reconnect guarantees, or live replay guarantees.

## Follow-up Decision

Classify the v0.4.5 in-flight gate as not passed:

- In-flight disconnect was not validated.
- The task completed before the disconnect could be performed.
- Completed-task recovery was validated again after a real transport closure.

A future stronger retry should make the long-running command non-interactively
permitted before starting Claude, then reconnect while that command is
confirmed active. Do not change product code based on this result; the observed
gap is in the validation task's Claude permission setup, not a demonstrated
AI Bridge release-blocking defect.
